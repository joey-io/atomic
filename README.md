# Atomic

A framework for building data systems. One shape. One interface.

```
{ guid, model, attr, lifecycle }
```

Everything is an atom — models, traits, hooks, config, tokens, and domain data.

`guid` is generated by Atomic and carries no business meaning.

`model` is a reference to the model atom defining:

- validation
- display
- permissions
- identity rules
- merge behavior
- retention
- indexes
- hooks
- generated surfaces

The system resolves behavior from the model atom itself.

Reserved model keys:

| Model | Purpose |
|------|------|
| `model` | Defines atom schemas and behavior |
| `trait` | Reusable field shapes |
| `report` | Derived views and query execution |
| `plugin` | Bundled capabilities |
| `hook` | Pipeline logic |
| `token` | Authentication |
| `config` | Cascading settings |
| `file` | Object storage pointers |
| `log` | Append-only audit entries |

Domain models — `person`, `facility`, `invoice`, `task` — are implementation-defined model atoms.

**Atomic** is the framework.
**atomic-crm** is a set of models, traits, reports, and plugins that make it a CRM.

---

## Principles

```
1. Everything is an atom
2. Everything flows through one interface
3. Everything is logged (same transaction, append-only)
4. Models drive behavior — not code
5. Surfaces are generated — not maintained
6. Plugins bundle capability — toggle on/off
7. Config cascades — workspace → tenant → system
8. Immutable atoms use corrections, not edits
9. Postgres is the platform — no external services beyond storage and email
10. Scale by adding instances — architecture unchanged
11. Fields are nullable — no migrations, UI renders what exists
12. Data is real-time — changes broadcast, clients subscribe
13. Identity is model-defined — not inferred from IDs
14. Creation is the record itself — mutation produces logs
15. References can be direct or query-resolved
```

## The Atom

```json
{
  "guid": "7b8f2f0c-5f0f-4a3d-9f0d-2d6e2d4d1c11",
  "model": {
    "$ref": "22222222-2222-4222-8222-222222222222"
  },
  "attr": {},
  "lifecycle": {}
}
```

| Field | Purpose |
|-------|---------|
| `guid` | Stable generated identity. No business meaning embedded. |
| `model` | Reference to the model atom defining schema and behavior. |
| `attr` | Domain data shaped by the model. |
| `lifecycle` | Interface-managed operational metadata. |

Atomic generates GUIDs internally.

GUIDs never encode:

- type
- tenant
- source
- business meaning

Those belong in data — not identity.

Real-world timestamps (`occurredAt`, `effectiveAt`, `filedAt`) belong in `attr`.

`lifecycle` tracks the atom inside Atomic.

## References

References are first-class primitives.

A reference can be:

| Form | Purpose |
|---|---|
| direct | point to a known atom GUID |
| query | resolve an atom by query |

### Direct Ref

```json
{
  "$ref": "22222222-2222-4222-8222-222222222222"
}
```

### Query Ref

```json
{
  "$ref": {
    "query": {
      "model": "model",
      "where": {
        "key": "invoice"
      },
      "limit": 1
    }
  }
}
```

Query refs are mainly used for:

- imports
- templates
- config
- portable manifests
- human-authored definitions

The interface resolves query refs into direct GUID refs before execution.

Most runtime operations use direct refs.

Traversal paths resolve through refs and nested JSON:

```txt
contribution.committee.treasurer.address.state
```

The same traversal language works across:

- filters
- reports
- GraphQL
- permissions
- hooks
- exports
- generated UI

## Traits

A trait is a reusable field shape.

```json
{
  "guid": "11111111-1111-4111-8111-111111111111",
  "model": {
    "$ref": "aaaaaaaa-aaaa-4aaa-8aaa-aaaaaaaaaaaa"
  },
  "attr": {
    "key": "address",
    "fields": {
      "street": { "kind": "text" },
      "city": { "kind": "text" },
      "state": { "kind": "text" },
      "zip": { "kind": "text" },
      "country": {
        "kind": "text",
        "default": "US"
      }
    }
  }
}
```

Traits are schema includes — not separate domain entities.

An address is usually embedded JSON, not its own atom.

Models reference traits:

```json
{
  "location": {
    "$ref": "11111111-1111-4111-8111-111111111111"
  }
}
```

One definition.
Many usages.

## Models

A model defines:

- fields
- validation
- display
- permissions
- identity
- merge rules
- retention
- hooks
- indexes
- generated surfaces

Example:

```json
{
  "guid": "22222222-2222-4222-8222-222222222222",
  "model": {
    "$ref": "bbbbbbbb-bbbb-4bbb-8bbb-bbbbbbbbbbbb"
  },
  "attr": {
    "key": "facility",
    "label": "Facility",
    "fields": {
      "name": {
        "kind": "text",
        "required": true
      },
      "location": {
        "$ref": "11111111-1111-4111-8111-111111111111"
      },
      "openedAt": {
        "kind": "datetime",
        "filterable": true,
        "sortable": true
      }
    },
    "identity": {
      "keys": [
        ["externalIds.fec"],
        ["name", "location.city", "location.state"]
      ]
    },
    "display": {
      "row": [
        "name",
        "location.city"
      ]
    },
    "behavior": {
      "mutable": true,
      "mergeStrategy": "model-defined"
    }
  }
}
```

Adding a new type means creating a model atom.

No code.
No deployment.

## Identity & Deduplication

Identity is model-defined.

Atomic never infers sameness from GUIDs.

Models define identity key sets:

```json
{
  "identity": {
    "keys": [
      ["externalIds.fec"],
      ["email"],
      ["name", "birthDate"]
    ]
  }
}
```

During writes/imports:

```txt
resolve model
  → validate
  → resolve identity
  → deduplicate / merge
  → execute
```

Possible outcomes:

| Result | Behavior |
|---|---|
| no match | create atom |
| confident match | merge/update |
| ambiguous | create review task |

---

## Merging

Merging is first-class.

```txt
A + B → A
```

Where:

- `A` becomes canonical
- refs to `B` resolve to `A`
- merge history remains queryable
- merge operations produce log atoms

Merges are append-only and auditable.

## Reports

Reports are atoms.

Reports define:

- derived views
- aggregations
- reusable queries
- resolution logic

Example:

```json
{
  "guid": "33333333-3333-4333-8333-333333333333",
  "model": {
    "$ref": "cccccccc-cccc-4ccc-8ccc-cccccccccccc"
  },
  "attr": {
    "key": "invoices.byQuarter",
    "source": "invoice",
    "groupBy": ["issuedAt.quarter"],
    "measures": [
      {
        "field": "amountCents",
        "op": "sum",
        "as": "totalCents"
      }
    ]
  }
}
```

Query refs are effectively reusable report execution.

Queries become data.

## The Interface

Everything flows through one pipeline:

```txt
request
  → authenticate
  → resolve tenant/workspace
  → resolve refs
  → resolve model
  → resolve config
  → validate
  → resolve identity
  → deduplicate / merge
  → run hooks
  → execute
  → write logs
  → enqueue async hooks
  → broadcast
  → respond
```

This is the only way data enters or leaves Atomic.

## Surfaces

Generated from models.

Never maintained separately.

| Surface | Purpose |
|---|---|
| GraphQL | Generated schema |
| SDK | Typed generated client |
| MCP | Generated AI tool definitions |
| Web App | Generated forms/tables/detail pages |

Change a model → all surfaces update.

---

## Hooks

Hooks are atoms executed during pipeline phases.

| Phase | Purpose |
|---|---|
| `pre:save` | validate/transform |
| `pre:read` | scope/filter |
| `post:save` | workflows/notifications |
| `post:read` | redaction/computed fields |

Async hooks enqueue transactionally.

```txt
System
  └─ Tenant
       └─ Workspace
            └─ Database
```

No shared tenant table.
No tenant_id column.

Isolation is physical.

```txt
workspace → tenant → system
```

## Auth

Authentication:

```txt
request
  → hash token
  → lookup token atom
  → load actor
  → merge roles
  → check permissions
```

No passwords.
No OAuth.
No JWT requirement.

## Lifecycle

Atoms have states:

| Status | Meaning |
|---|---|
| active | normal |
| archived | hidden but queryable |
| deleted | removed |

Immutable data uses corrections/reversals.

---

## Logs

Every mutation produces append-only log atoms.

Creation is represented by the atom itself.

Mutations produce log atoms:

```txt
create → atom exists
update → log atom
merge → log atom
archive → log atom
restore → log atom
```

Logs contain:

- affected atom
- action
- before
- after
- actor
- timestamp
- source

Logs are queryable like all other atoms.

---

## Database

Each workspace database contains:

- atoms
- logs

Atoms:

```txt
guid
model
attr JSONB
lifecycle JSONB
```

Logs:

```txt
guid
atom_guid
action
before JSONB
after JSONB
actor
occurred_at
```

Accelerators may be added:

- GIN indexes
- generated columns
- trigram indexes
- materialized reports
- graph accelerators
- identity indexes

All accelerators are derived.
Never source-of-truth.
