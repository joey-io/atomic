# Atomic

A framework for building data systems. One shape. One interface.

```
{ guid, model, attr, lifecycle }
```

Everything is an atom — models, traits, queries, hooks, config, tokens, and domain data.

`guid` is generated by Atomic and carries no business meaning.

`model` is a reference to the model atom defining:

- validation
- display
- permissions
- identity rules
- merge behavior
- retention
- indexes
- hooks
- generated surfaces

The system resolves behavior from the model atom itself.

Reserved model keys:

| Model | Purpose |
|------|------|
| `model` | Defines atom schemas and behavior |
| `trait` | Reusable field shapes |
| `query` | Reusable query, resolver, derived view, and index intent |
| `plugin` | Bundled capabilities |
| `hook` | Pipeline logic |
| `token` | Authentication |
| `config` | Cascading settings |
| `file` | Object storage pointers |
| `log` | Append-only audit entries |

Domain models — `person`, `facility`, `invoice`, `task` — are implementation-defined model atoms.

**Atomic** is the framework.
**atomic-crm** is a set of models, traits, queries, and plugins that make it a CRM.

---

## Principles

```
1. Everything is an atom
2. Everything flows through one interface
3. Everything is logged (same transaction, append-only)
4. Models drive behavior — not code
5. Surfaces are generated — not maintained
6. Plugins bundle capability — toggle on/off
7. Config cascades — workspace → tenant → system
8. Immutable atoms use corrections, not edits
9. Postgres is the platform — no external services beyond storage and email
10. Scale by adding instances — architecture unchanged
11. Fields are nullable — no migrations, UI renders what exists
12. Data is real-time — changes broadcast, clients subscribe
13. Identity is model-defined — not inferred from IDs
14. Creation is the record itself — mutation produces logs
15. References are direct refs or query-backed refs
16. Queries define reusable access patterns and indexing intent
```

## The Atom

```json
{
  "guid": "7b8f2f0c-5f0f-4a3d-9f0d-2d6e2d4d1c11",
  "model": {
    "$ref": "22222222-2222-4222-8222-222222222222"
  },
  "attr": {},
  "lifecycle": {}
}
```

| Field | Purpose |
|-------|---------|
| `guid` | Stable generated identity. No business meaning embedded. |
| `model` | Reference to the model atom defining schema and behavior. |
| `attr` | Domain data shaped by the model. |
| `lifecycle` | Interface-managed operational metadata. |

Atomic generates GUIDs internally.

GUIDs never encode:

- type
- tenant
- source
- business meaning

Those belong in data — not identity.

Real-world timestamps (`occurredAt`, `effectiveAt`, `filedAt`) belong in `attr`.

`lifecycle` tracks the atom inside Atomic.

## References

References are first-class primitives.

A reference can be direct:

```json
{
  "$ref": "22222222-2222-4222-8222-222222222222"
}
```

Or query-backed:

```json
{
  "$ref": {
    "query": "33333333-3333-4333-8333-333333333333",
    "params": {
      "key": "invoice"
    }
  }
}
```

A query-backed ref means:

```txt
Run this query atom with these params.
Use the returned atom as the reference.
```

Most runtime operations use direct refs.

Query-backed refs are used when the target must be resolved by reusable query logic, such as:

- imports
- templates
- config
- portable manifests
- human-authored definitions
- identity resolution
- relationship resolution

The interface resolves query-backed refs into direct GUID refs before execution.

Traversal paths resolve through refs and nested JSON:

```txt
contribution.committee.treasurer.address.state
```

The same traversal language works across:

- filters
- queries
- GraphQL
- permissions
- hooks
- exports
- generated UI

## Traits

A trait is a reusable field shape.

```json
{
  "guid": "11111111-1111-4111-8111-111111111111",
  "model": {
    "$ref": "aaaaaaaa-aaaa-4aaa-8aaa-aaaaaaaaaaaa"
  },
  "attr": {
    "key": "address",
    "fields": {
      "street": { "kind": "text" },
      "city": { "kind": "text" },
      "state": { "kind": "text" },
      "zip": { "kind": "text" },
      "country": {
        "kind": "text",
        "default": "US"
      }
    }
  }
}
```

Traits are schema includes — not separate domain entities.

An address is usually embedded JSON, not its own atom.

Models reference traits:

```json
{
  "location": {
    "$ref": "11111111-1111-4111-8111-111111111111"
  }
}
```

One definition.
Many usages.

## Models

A model defines:

- fields
- validation
- display
- permissions
- identity
- merge rules
- retention
- hooks
- indexes
- generated surfaces

Example:

```json
{
  "guid": "22222222-2222-4222-8222-222222222222",
  "model": {
    "$ref": "bbbbbbbb-bbbb-4bbb-8bbb-bbbbbbbbbbbb"
  },
  "attr": {
    "key": "facility",
    "label": "Facility",
    "fields": {
      "name": {
        "kind": "text",
        "required": true
      },
      "location": {
        "$ref": "11111111-1111-4111-8111-111111111111"
      },
      "openedAt": {
        "kind": "datetime",
        "filterable": true,
        "sortable": true
      }
    },
    "identity": {
      "keys": [
        ["externalIds.fec"],
        ["name", "location.city", "location.state"]
      ]
    },
    "display": {
      "row": [
        "name",
        "location.city"
      ]
    },
    "behavior": {
      "mutable": true,
      "mergeStrategy": "model-defined"
    }
  }
}
```

Adding a new type means creating a model atom.

No code.
No deployment.

## Identity & Deduplication

Identity is model-defined.

Atomic never infers sameness from GUIDs.

Models define identity key sets:

```json
{
  "identity": {
    "keys": [
      ["externalIds.fec"],
      ["email"],
      ["name", "birthDate"]
    ]
  }
}
```

During writes/imports:

```txt
resolve model
  → validate
  → resolve identity
  → deduplicate / merge
  → execute
```

Possible outcomes:

| Result | Behavior |
|---|---|
| no match | create atom |
| confident match | merge/update |
| ambiguous | create review task |

## Merging

Merging is first-class.

```txt
A + B → A
```

Where:

- `A` becomes canonical
- refs to `B` resolve to `A`
- merge history remains queryable
- merge operations produce log atoms

Merges are append-only and auditable.

## Queries

Queries are atoms.

A query defines reusable access logic:

- list queries
- detail queries
- pivots
- aggregations
- relationship resolvers
- reference resolvers
- search definitions
- materialized views
- derived dashboards
- index intent

Example:

```json
{
  "guid": "33333333-3333-4333-8333-333333333333",
  "model": {
    "$ref": "cccccccc-cccc-4ccc-8ccc-cccccccccccc"
  },
  "attr": {
    "key": "invoices.byQuarter",
    "source": {
      "$ref": "22222222-2222-4222-8222-222222222222"
    },
    "where": {
      "issuedAt": {
        "between": ["{{start}}", "{{end}}"]
      }
    },
    "groupBy": ["issuedAt.quarter"],
    "measures": [
      {
        "field": "amountCents",
        "op": "sum",
        "as": "totalCents"
      }
    ],
    "returns": "rows",
    "indexing": {
      "paths": [
        "issuedAt",
        "amountCents"
      ],
      "strategy": "materialize-when-expensive"
    }
  }
}
```

Queries are also the clean way to express query-backed refs.

Example: resolve a model by key.

```json
{
  "guid": "44444444-4444-4444-8444-444444444444",
  "model": {
    "$ref": "cccccccc-cccc-4ccc-8ccc-cccccccccccc"
  },
  "attr": {
    "key": "models.byKey",
    "source": {
      "$ref": "bbbbbbbb-bbbb-4bbb-8bbb-bbbbbbbbbbbb"
    },
    "where": {
      "key": "{{key}}"
    },
    "limit": 1,
    "returns": "atom",
    "indexing": {
      "unique": ["key"],
      "paths": ["key"],
      "strategy": "btree"
    }
  }
}
```

Then a ref can use that query:

```json
{
  "$ref": {
    "query": "44444444-4444-4444-8444-444444444444",
    "params": {
      "key": "invoice"
    }
  }
}
```

This keeps the primitive small:

```txt
direct ref → known atom
query-backed ref → query atom returns atom
```

No separate lookup language is needed.

Queries become data.

### Query Indexing

The global `query` model defines how query atoms express indexing intent.

Indexing is not manually maintained application code.

Indexing is derived from query atoms.

A query can declare:

| Attribute | Purpose |
|---|---|
| `indexing.paths` | JSON paths repeatedly filtered, sorted, grouped, joined, or traversed |
| `indexing.unique` | paths that should be unique for the query source |
| `indexing.strategy` | preferred physical strategy: `btree`, `gin`, `trigram`, `vector`, `materialized`, or `auto` |
| `indexing.materialize` | whether results should be persisted/rebuilt |
| `indexing.refresh` | refresh policy for materialized query results |
| `indexing.derivedEdges` | ref paths worth projecting into graph accelerators |

Example:

```json
{
  "indexing": {
    "paths": [
      "contributor",
      "committee",
      "electionCycle",
      "amountCents",
      "receivedAt"
    ],
    "unique": [
      "externalIds.fecTransactionId"
    ],
    "strategy": "auto",
    "materialize": false,
    "derivedEdges": [
      "contributor",
      "committee"
    ]
  }
}
```

Atomic may create derived accelerators from query intent:

- btree indexes
- GIN indexes
- generated columns
- trigram search indexes
- vector indexes
- materialized query tables
- graph edge tables

All accelerators are derived.
All are rebuildable.
None are source-of-truth.

## The Interface

Everything flows through one pipeline:

```txt
request
  → authenticate
  → resolve tenant/workspace
  → resolve refs
  → resolve model
  → resolve config
  → validate
  → resolve identity
  → deduplicate / merge
  → plan query / accelerators
  → run hooks
  → execute
  → write logs
  → enqueue async hooks
  → broadcast
  → respond
```

This is the only way data enters or leaves Atomic.

## Surfaces

Generated from models and queries.

Never maintained separately.

| Surface | Purpose |
|---|---|
| GraphQL | Generated schema and query fields |
| SDK | Typed generated client |
| MCP | Generated AI tool definitions |
| Web App | Generated forms/tables/detail pages/dashboards |

Change a model or query → all surfaces update.

---

## Hooks

Hooks are atoms executed during pipeline phases.

| Phase | Purpose |
|---|---|
| `pre:save` | validate/transform |
| `pre:read` | scope/filter |
| `post:save` | workflows/notifications |
| `post:read` | redaction/computed fields |

Async hooks enqueue transactionally.

## Lifecycle

Atoms have states:

| Status | Meaning |
|---|---|
| active | normal |
| archived | hidden but queryable |
| deleted | removed |

## Logs

Every mutation produces append-only log atoms.

Creation is represented by the atom itself.

Mutations produce log atoms:

```txt
create → atom exists
update → log atom
merge → log atom
archive → log atom
restore → log atom
```

Logs contain:

- affected atom
- action
- before
- after
- actor
- timestamp
- source

Logs are queryable like all other atoms.
